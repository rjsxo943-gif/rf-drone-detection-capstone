# 2026-06-05 RF 드론 탐지 및 AoA 실험·수정 보고서

## 1. 실험 및 수정 목적

본 작업의 목적은 RF 드론 탐지 시스템에서 드론 신호 검출 후 AoA(Angle of Arrival) 계산이 불안정하게 표시되는 문제를 개선하고, 실외 2.4GHz 대역 환경에서 발생하는 Wi-Fi 및 기타 RF 간섭에 대한 시스템 동작을 분석하는 것이다.

기존 시스템은 CNN voting 결과가 `confirmed=True`가 되면 AoA 계산을 수행하였다. 그러나 실험 중 현재 selected block이 육안상 드론 패턴으로 보이지 않거나 CNN raw 결과가 NotDrone에 가까운 경우에도, 이전 voting 상태가 유지되어 AoA가 계속 계산되는 현상이 관찰되었다. 또한 coherence가 낮은 block에서도 AoA가 valid처럼 표시되어, 실제 각도 추정값이 60도, 90도 또는 반대 방향으로 튀는 문제가 있었다.

이에 따라 오늘은 다음 두 가지를 중심으로 개선하였다.

1. AoA 계산/표시 조건을 YAML 기반 gate로 분리
2. OpenCV 및 CLI 출력 가독성 개선

---

## 2. 기존 문제점

### 2.1 CNN voting confirmed 유지로 인한 AoA 오표시

기존 구조는 다음과 같았다.

```text
selected block 선택
→ CNN inference
→ CNN temporal voting confirmed=True
→ AoA 계산
```

이 구조에서는 현재 block의 CNN raw 결과가 NotDrone이어도, 이전 voting history에 의해 `confirmed=True`가 유지되면 AoA가 계속 계산될 수 있었다.

즉 다음과 같은 문제가 발생하였다.

```text
현재 block: CNN raw = NotDrone
이전 voting history: Drone confirmed 유지
결과: AoA 계산됨
```

이는 AoA 계산 자체의 수식 오류라기보다는, 전단 CNN/voting 상태가 후단 AoA 표시 조건으로 그대로 전달된 문제로 볼 수 있다.

### 2.2 낮은 coherence 구간에서의 AoA 튐

AoA는 두 수신 채널 간 위상차를 기반으로 계산된다. 따라서 두 채널의 위상 관계가 안정적이어야 한다. 그러나 coherence가 낮은 block에서도 AoA가 표시되면 다음과 같은 현상이 발생할 수 있다.

```text
coherence 낮음
→ 위상차 불안정
→ angle 값이 갑자기 60도, 90도, 반대각으로 튐
```

따라서 AoA 결과는 CNN confirmed 여부뿐 아니라, 현재 block의 CNN raw 결과, confidence, coherence를 함께 고려해야 한다.

## 3. 오늘 적용한 코드 수정 사항

### 3.1 `configs/ui.yaml`에 AoA gate 설정 추가

`configs/ui.yaml`의 `live_rf_viewer` 섹션 아래에 `aoa_gate` 설정을 추가하였다.

```yaml
live_rf_viewer:
  blocks_per_update: 20
  select_policy: raw_gate_pass_score_max
  cli_log_every_n: 1
  disable_cli_log: false

  aoa_gate:
    enabled: true
    require_voting_confirmed: true
    require_current_drone: true
    min_current_confidence: 0.90
    min_coherence: 0.90
    display_only_valid: true
    show_skip_reason: true
```

각 옵션의 의미는 다음과 같다.

| 설정                         | 의미                                             |
| -------------------------- | ---------------------------------------------- |
| `enabled`                  | AoA gate 기능 전체 ON/OFF                          |
| `require_voting_confirmed` | CNN temporal voting confirmed 상태일 때만 AoA 허용    |
| `require_current_drone`    | 현재 selected block의 CNN raw 결과도 Drone이어야 AoA 허용 |
| `min_current_confidence`   | 현재 CNN confidence 최소 기준                        |
| `min_coherence`            | AoA 표시를 허용할 최소 coherence                       |
| `display_only_valid`       | valid한 AoA만 화면에 표시                             |
| `show_skip_reason`         | AoA가 막힌 이유를 CLI/overlay에 표시                    |

이 설정을 통해 detection confirmed와 AoA 표시 조건을 분리하였다.

---

### 3.2 AoA pre-gate 추가

`live_rf_viewer_drone_aoa.py`에 `check_aoa_pre_gate()` 함수를 추가하였다.

이 함수는 AoA 계산 전에 다음 조건을 확인한다.

```text
1. raw gate가 AoA를 막지 않는가?
2. CNN temporal voting confirmed=True인가?
3. 현재 selected block의 CNN raw class가 Drone인가?
4. 현재 CNN confidence가 YAML 기준 이상인가?
```

수정 후에는 다음 상황에서 AoA가 계산되지 않는다.

```text
CNN raw = NotDrone
smooth = Drone
confirmed = True
→ AOA skip=current_not_drone
```

즉, 이전 voting 상태만으로 AoA가 열리는 문제를 줄였다.

---

### 3.3 AoA post-gate 추가

AoA 계산 후에는 `apply_aoa_post_gate()`를 통해 coherence를 검사하도록 수정하였다.

AoA는 계산을 해야 coherence 값을 얻을 수 있으므로, 먼저 AoA를 계산한 뒤 coherence가 낮으면 다음과 같이 처리한다.

```text
coherence < min_coherence
→ aoa_valid = False
→ aoa_skipped_reason = low_coherence
→ angle 표시 무효화
```

예상 출력은 다음과 같다.

```text
[AOA] skip=low_coherence:0.367<0.900
```

이를 통해 위상 안정성이 낮은 block에서 발생하는 각도 튐을 화면에서 억제할 수 있다.

### 3.4 신호발생기 실험용 AoA-only 모드 가능

기존에는 신호발생기 각도 확인용으로 다음 명령어를 사용하였다.

```bash
PYTHONPATH=. python scripts/live_rf_viewer.py --mode full
```

이제는 최신 viewer를 사용하되, YAML에서 CNN 기반 AoA gate를 풀면 신호발생기 AoA 확인도 가능하다.

```bash
PYTHONPATH=. python scripts/live_rf_viewer_drone_aoa.py --mode full
```

신호발생기용 설정 예시는 다음과 같다.

```yaml
aoa_gate:
  enabled: true
  require_voting_confirmed: false
  require_current_drone: false
  min_current_confidence: 0.00
  min_coherence: 0.90
  display_only_valid: true
  show_skip_reason: true
```

이 경우 CNN Drone 판정은 요구하지 않고, coherence 기준만으로 AoA 표시 여부를 판단한다. 따라서 신호발생기처럼 드론 CNN 판정이 필요 없는 실험에서도 최신 viewer 하나로 AoA를 확인할 수 있다.

### 3.5 포화 시 gain 조절 안내 문구 추가

기존 viewer에서는 raw signal이 overload 상태일 때 다음과 같은 안내 문구를 사용하였다.

```text
OVERLOAD → TRY_LOWER_GAIN
```

최신 `live_rf_viewer_drone_aoa.py`에서도 포화가 감지되면 OpenCV overlay 및 CLI에 다음 문구를 표시하도록 개선하였다.

```text
[WARN] OVERLOAD -> TRY_LOWER_GAIN
```

또는 CLI에서는 다음과 같이 표시된다.

```text
[WARNING] status=OVERLOAD suggestion=TRY_LOWER_GAIN
```

이를 통해 실험 중 수신 신호가 너무 강해 포화될 경우, 사용자가 즉시 gain을 낮춰야 함을 확인할 수 있다.

### 3.6 CLI 출력 가독성 개선

기존 CLI 출력은 RF, raw feature, raw gate, CNN, AoA 정보가 한 줄에 모두 붙어서 표시되어 읽기 어려웠다.

기존 예시는 다음과 같았다.

```text
[RF] idx=... mode=... gain=... sel=... raw_p99=... RAW_GATE=... CNN=... AOA=...
```

수정 후에는 update 1회당 다음과 같이 그룹별로 줄을 나누어 출력한다.

```text
[RF] idx=23 mode=full gain=40.0 cf=2450.000MHz sr=5.000MS/s
[RAW] sel=7/20 policy=raw_gate_pass_score_max raw_p99=... rms=... max=... fp99=... overload=False
[RAW_GATE] RAW_GATE_PASS pass=True score=... noise=... thr=...
[CAL] thr=... raw=SAFE sat=0.000%
[CNN] raw=Drone conf=0.982
[CNN_VOTE] smooth=Drone avg=0.944 votes=4/3 confirmed=True
[AOA] angle=-18.2deg valid=True coh=0.973 phase=...
```

이를 통해 실시간 디버깅 시 어느 단계에서 문제가 발생했는지 쉽게 확인할 수 있게 되었다.

### 3.7 OpenCV overlay compact layout 적용

OpenCV 오른쪽 패널은 기존에 정보가 너무 많이 표시되어 AoA가 CNN 아래로 밀리고, 패널 밖으로 잘려 보이지 않는 문제가 있었다.

OpenCV renderer 구조상 오른쪽 패널은 제한된 폭과 줄 높이를 가지며, 긴 줄은 자동 줄바꿈될 수 있다. 따라서 단순히 AoA 순서만 위로 올리는 것은 충분하지 않았다.

이에 따라 OpenCV overlay는 다음과 같은 compact 구조로 변경하였다.

```text
[RF] idx=... LIVE full
cf=2450.000M sr=5.000M g=40.0
[RAW] sel=... p99=... rms=...
max=... fp99=... ov=False
[WARN] OVERLOAD -> TRY_LOWER_GAIN
[AOA] angle=... valid=... coh=...
phase=...
[CNN] Drone conf=...
[VOTE] Drone 4/3 ok=True
[GATE] ...
[CAL] ...
```

OpenCV overlay는 현장 확인용 핵심 정보만 표시하고, 상세 정보는 CLI와 CSV에 남기는 방식으로 역할을 분리하였다.

## 5. 실험 결과

### 5.1 신호발생기 기반 AoA 검증

신호발생기를 사용하여 AoA를 검증하였다.

실험 조건은 다음과 같다.

| 항목     | 조건        |
| ------ | --------- |
| 중심 주파수 | 약 2.45GHz |
| Gain   | 40        |
| 거리     | 10m       |
| 신호원    | 신호발생기     |

실험 결과, gain=40 및 거리 10m 조건에서 약 ±65도까지 각도 측정이 가능하였다.

신호발생기는 드론 조종기보다 신호가 안정적이고 단일 톤에 가까우므로, 위상차 기반 AoA가 비교적 안정적으로 동작하였다. 따라서 신호발생기 조건에서는 넓은 각도 범위에서 AoA 추정이 가능함을 확인하였다.

정리하면 다음과 같다.

```text
신호발생기 조건:
gain=40, 거리=10m
→ 약 ±65도까지 AoA 측정 가능
```

### 5.2 드론 조종기 기반 AoA 검증

드론 조종기를 사용하여 실제 RF 신호 조건에서 AoA를 검증하였다.

특히 Y편파 안테나를 사용했을 때, 조종기를 세운 방향에서 신호가 약 2~3배 강하게 수신되었다. 이는 조종기 송신 편파와 수신 안테나 편파가 정합되었기 때문으로 해석할 수 있다.

각도기 기준과 위상차 기반 AoA 결과는 다음과 같았다.

| 각도기 기준 | AoA 추정값 |       오차 |
| -----: | ------: | -------: |
|   -15° |    -18° |    약 -3° |
|   -45° |    -48° |    약 -3° |
|   -60° |    -70° |   약 -10° |
|     0° |     -1° |    약 -1° |
|    15° |     18° |    약 +3° |
|    30° |     28° |    약 -2° |
|    40° |  37~43° | 약 -3~+3° |
|    58° |     57° |    약 -1° |

대부분의 구간에서 각도기 기준값과 AoA 추정값이 잘 일치하였다. 특히 ±40도 이내에서는 오차가 대체로 ±1~3도 수준으로 안정적이었다.

다만 -60도 부근에서는 약 10도 정도의 오차가 발생하였고, ±65도까지 측정하려고 할 경우 90도 또는 반대 각도로 튀는 현상이 관찰되었다.

따라서 드론 조종기 실신호 기준 안정 AoA 측정 범위는 보수적으로 ±55~60도 정도로 보는 것이 타당하다.

```text
드론 조종기 조건:
Y편파 안테나 사용 시 수신 세기 2~3배 증가
±55~60도 범위에서는 비교적 안정적
±65도 이상에서는 각도 튐 발생
```

## 6. 관찰된 문제 및 원인 분석

### 6.1 실외 Wi-Fi 간섭

실외 환경임에도 2.4GHz 대역에서 Wi-Fi 또는 기타 RF 신호가 강하게 들어오는 현상이 있었다.

2.4GHz ISM 대역은 Wi-Fi, Bluetooth, 드론 조종기, 기타 무선 장비가 함께 사용하는 대역이다. 따라서 실외라고 해도 주변 AP, 휴대폰 핫스팟, 블루투스 장비, 건물 내부 Wi-Fi 신호가 유입될 수 있다.

실험 중 다음과 같은 흐름이 가능하였다.

```text
Wi-Fi burst 발생
→ raw score 증가
→ representative block으로 선택
→ CNN이 Drone-like로 오판
→ voting confirmed 유지
→ AoA 계산
```

이 문제는 단순한 AoA 수식 오류가 아니라, RF 간섭 + CNN false positive + voting 유지 구조가 결합된 문제로 볼 수 있다.

---

### 6.2 multi-frequency 모델의 false positive 가능성

최근 업데이트한 모델은 2.450GHz뿐만 아니라 2.460GHz 및 2.465GHz 데이터를 추가한 모델이었다. 이로 인해 드론 신호의 주파수 일반화 성능은 좋아질 수 있지만, Drone-like로 판단하는 범위도 넓어져 일부 Wi-Fi burst 또는 다른 RF 신호를 Drone-like로 오판할 가능성이 증가할 수 있다.

따라서 모델은 용도에 따라 분리해서 사용하는 것이 적절하다.

```text
2.45GHz precision model:
- 오탐 억제에 유리
- 시연/보고서용 안정 모드에 적합

2.450/2.460/2.465GHz freqboost model:
- 주파수 일반화에 유리
- field coverage 확장용
- Wi-Fi hard negative 보강 필요
```

## 7. 현재 시스템 동작 정리

현재 최신 viewer의 전체 흐름은 다음과 같다.

```text
N개 block 수집
→ 각 block raw feature 계산
→ 각 block RawNoiseGate 평가
→ raw gate 통과 block 중 score_max가 가장 큰 대표 block 선택
→ selected block을 CNN voting에 입력
→ selected block에 대해 AoA gate 평가
→ 조건 만족 시 AoA 계산 및 표시
```

정밀모드의 핵심인 `blocks_per_update: 20` 및 `select_policy: raw_gate_pass_score_max`는 유지된다. 즉 이번 수정은 정밀모드를 해제한 것이 아니라, selected block 이후의 AoA 계산/표시 조건만 개선한 것이다.

## 8. 향후 개선 방향

### 8.1 Wi-Fi hard negative 데이터 추가

현재 false positive의 주요 원인 중 하나는 Wi-Fi 및 기타 2.4GHz 간섭 신호가 Drone-like로 오분류되는 것이다. 따라서 Wi-Fi hard negative 데이터를 추가 수집하여 모델이 드론 조종기 burst와 Wi-Fi burst를 더 잘 구분하도록 개선할 필요가 있다.

### 8.2 주파수별 threshold 분리

2.450GHz, 2.460GHz, 2.465GHz에서 신호 패턴과 주변 간섭 환경이 다를 수 있으므로, 주파수별 CNN threshold 또는 raw gate threshold를 분리하는 것이 필요하다.

### 8.3 AoA gate 모드 분리

YAML 기반으로 다음과 같은 모드를 운용할 수 있다.

#### 시연/보고서 안정 모드

```yaml
aoa_gate:
  enabled: true
  require_voting_confirmed: true
  require_current_drone: true
  min_current_confidence: 0.90
  min_coherence: 0.90
  display_only_valid: true
  show_skip_reason: true
```

#### 신호발생기 AoA 확인 모드

```yaml
aoa_gate:
  enabled: true
  require_voting_confirmed: false
  require_current_drone: false
  min_current_confidence: 0.00
  min_coherence: 0.90
  display_only_valid: true
  show_skip_reason: true
```

이렇게 모드를 분리하면, 하나의 최신 viewer에서 드론 조종기 실험과 신호발생기 AoA 검증을 모두 수행할 수 있다.

## 9. 결론

오늘 실험에서는 신호발생기 조건에서 gain=40, 거리 10m 기준 약 ±65도까지 AoA 측정이 가능함을 확인하였다. 드론 조종기 실신호 조건에서는 Y편파 안테나 사용 시 수신 신호가 2~3배 강해졌으며, 각도기 기준값과 위상차 기반 AoA 결과가 대부분 잘 일치하였다. 다만 ±65도 부근에서는 phase wrapping, coherence 저하, 멀티패스 영향으로 인해 각도가 90도 또는 반대 방향으로 튀는 현상이 발생하였다. 따라서 드론 조종기 실신호 기준 안정 측정 범위는 ±55~60도 정도로 보는 것이 타당하다.

코드 측면에서는 `configs/ui.yaml`에 AoA gate 설정을 추가하고, `live_rf_viewer_drone_aoa.py`에서 AoA pre-gate와 post-gate를 구현하였다. 이를 통해 CNN voting confirmed 상태가 유지되더라도 현재 selected block이 NotDrone이면 AoA를 막고, coherence가 낮은 경우에도 AoA 표시를 무효화할 수 있게 되었다.

또한 CLI 출력은 RF, RAW, RAW_GATE, CAL, CNN, AOA 그룹으로 나누어 가독성을 개선하였고, OpenCV overlay는 compact layout으로 수정하여 AoA가 화면 아래로 밀려 보이지 않는 문제를 완화하였다. 포화 상황에서는 `TRY_LOWER_GAIN` 안내문구를 표시하도록 하여 현장 실험 중 gain 조절 판단을 쉽게 하였다.

종합적으로, 오늘 수정은 드론 탐지 성능 자체를 바꾸기보다는, 탐지 결과와 AoA 표시 조건을 분리하여 실험 신뢰성과 현장 가독성을 개선한 작업이다.
